# codex-forge V3 规划

## 背景

V2 已经完成多 Agent CLI 的基础闭环：

- `plan` 会生成用户可读的 todo 清单，并派生内部 `ExecutionGraph`
- `run` 可以复用最近一次匹配的 `plan` 结果
- commander 已具备 `规划 → 分发 → reviewer gate → apply → verification → summary` 的主流程
- session 工件、worker handoff、integration 验证和 replay 已基本可用

这次真实 `run` 暴露出一个很关键的现实问题：**系统已经能跑通，但“可相信到什么程度”仍然不够精细**。  
尤其在以下方面，V2 还存在明显缺口：

- provider 结构化调用失败时，系统虽然可以绕过，但缺少更统一的鲁棒性设计
- `plan` 与 `run` 虽然已联动，但“复用什么、信任什么、何时降级”还不够显式
- reviewer 可以放行，但对“改动范围漂移”的约束仍偏弱
- worker handoff、reviewer 结论、final summary 之间偶尔会出现信息不完全一致
- live smoke 受环境限制时，只能靠人工解释，系统没有把“环境原因”和“代码失败”清晰分层

V3 的目标不是继续堆更多 agent 花活，而是把系统从“能跑”升级到“**稳定、可信、可解释、可收敛**”。

---

## V3 目标

### 1. 让 `plan` / `run` 联动成为一等能力

V3 要把“先规划，再执行”从体验层能力升级成架构层约束：

- `run` 默认优先绑定最近一次匹配的 `plan`
- `plan` 输出不只是 todo，还要成为后续执行的**显式契约**
- `run` 必须知道哪些内容来自 plan，哪些内容是执行期新增推断
- 若执行结果偏离 plan，系统要给出结构化偏差说明，而不是只在最终摘要里口头提醒

### 2. 让自动收敛建立在“可信变更”而不是“存在 patch”上

V2 的 auto-safe 更接近“patch 能不能应用”；V3 要升级为“**这份 patch 值不值得信任**”：

- 自动应用前必须判断改动是否越界
- worker 自报 `touched_files` 不能被直接信任，必须和真实 diff 交叉验证
- reviewer 的放行不能只表示“没看到大问题”，还要明确回答“是否符合 plan 范围”
- summary 不能同时出现“reviewer 放行”和“整体只部分可信”却没有给出具体分层原因

### 3. 让验证体系具备“环境感知”

本次博客案例中，真实 live smoke 因沙箱不允许绑定端口失败。  
V3 应把这类情况系统化处理：

- 区分“代码失败”和“环境受限”
- 在环境受限时，自动降级到等价验证
- 在最终摘要里把“已验证 / 未验证 / 因环境未验证”明确拆开
- 不再让 reviewer、tester、summary 各写一套近似但不完全一致的表述

### 4. 让最终摘要成为真正可执行的收敛结论

V2 的最终摘要已经能聚合信息，但还不够“可落地”：

- 需要明确列出“建议保留的文件”
- 需要明确列出“建议人工复核的文件”
- 需要明确列出“当前可直接接收的结论”和“当前仍需人工判断的结论”
- 需要能够把这份结果直接转成下一步动作，而不是停留在长文本描述

---

## V3 非目标

V3 暂不优先做以下事情：

- 不以“远程分布式 worker”作为本阶段主目标
- 不优先增加更多角色模板或更复杂的 agent 人设
- 不优先做前端 UI 或 Web 控制台
- 不优先解决所有 provider 的通用联网问题；优先解决 codex-forge 内部的鲁棒性和收敛逻辑

---

## 核心改动提案

## 一、执行契约升级：从 `ExecutionGraph` 到 `ExecutionContract`

V2 已有 `ExecutionGraph`，但 V3 需要把它升级成更强约束的执行契约。

### 目标

让 plan 阶段产物不仅描述“要做什么”，还描述：

- 允许改哪些文件
- 不允许改哪些边界文件
- 每个节点的可接受输出范围
- 每个节点最小验证要求
- 哪些偏差属于可接受实现差异，哪些偏差必须触发警告或降级

### 建议新增字段

- `allowed_paths`
- `forbidden_paths`
- `expected_artifacts`
- `required_verifications`
- `scope_guard`
- `drift_policy`

### 价值

- architect 的边界约束不再只存在于自然语言 handoff
- implementer 的“我没越界”和系统看到的真实 diff 可以对齐
- reviewer 可以按契约做 gate，而不是凭经验做只读判断

---

## 二、范围漂移检测：内建变更可信度评分

本次 run 最值得进入 V3 的能力，就是“范围漂移检测”。

### 需要解决的问题

- implementer 自报改动范围可能过大
- 实际 diff、handoff 描述、reviewer 结论、summary 可能不一致
- 系统目前只能在最终阶段以文字提醒“需要人工复核”

### V3 实现方向

在 apply 前引入 `ChangeTrustReport`：

- 对比 `plan` 契约中的 `allowed_paths` / `forbidden_paths`
- 对比 worker handoff 的 `touched_files`
- 对比真实 diff 文件集合
- 对比 reviewer 结论是否显式回应范围问题

### 输出结果建议

- `trust_level = high | medium | low`
- `scope_drift = none | minor | major`
- `safe_to_auto_apply = true | false`
- `human_review_required_files`
- `out_of_scope_files`

### 收敛策略

- `high`：允许进入 auto-safe
- `medium`：允许只应用白名单文件
- `low`：强制降级为 bundle 或人工确认

---

## 三、apply 阶段升级：支持“部分接收”

V2 的 apply 更像“整包应用或整包降级”。  
V3 应支持更贴近真实工程的“部分接收”。

### 新能力

- 基于路径白名单只接收博客核心文件
- 自动剔除超范围改动
- 输出“已接收文件 / 已拒绝文件 / 待人工复核文件”
- 在 `apply-mode none` 时也生成一份“建议接收清单”

### 直接收益

这次博客案例里，V3 理想行为应该是：

- 自动识别博客核心文件为候选接收项
- 自动把与博客目标无关的 CLI 核心文件列入人工复核
- reviewer 即便放行，也不代表整包全收

### 建议产物

- `integration/accepted-files.json`
- `integration/rejected-files.json`
- `integration/manual-review-files.json`

---

## 四、验证系统升级：显式区分“代码失败”和“环境受限”

本次 live smoke 缺失不是实现逻辑已被证伪，而是执行环境限制了本地端口监听。  
V3 应把这种信息结构化。

### 新模型

每条验证结果都应该带状态：

- `passed`
- `failed`
- `blocked_by_environment`
- `skipped`

### 行为要求

- 当检测到端口绑定、网络访问、权限、沙箱目录等环境限制时，不归类为普通失败
- 自动尝试等价验证路径
- summary 中单独输出“因环境未验证项”

### 需要沉淀的能力

- 端口监听受限识别
- 网络访问受限识别
- worktree 空壳 / unborn 分支识别
- artifact lock / 并发编译锁识别

### 目标

最终让系统能说清楚：

- 哪些功能被证实可用
- 哪些功能被证实有 bug
- 哪些功能当前无法验证，但不是因为代码已经失败

---

## 五、worktree 体系加固：解决空 worktree / unborn 分支问题

本次 tester 与 reviewer 都遇到了 worktree 空壳、`HEAD` 不存在、需要转向上游快照的问题。  
这说明 V2 的 worktree 初始化仍不够稳。

### V3 要求

- worker worktree 创建后必须经过“可用性自检”
- 若仓库处于 unborn 状态，系统要自动选择兼容策略
- reviewer / tester 不应再被迫手动绕到 implementer 快照

### 需要新增的机制

- worktree materialization check
- base revision 可用性检查
- 空树仓库兼容模式
- reviewer / tester 的只读快照挂载机制

### 验收标准

- reviewer 和 tester 拿到的工作目录应当天然可读可审，不依赖临时绕路

---

## 六、provider 鲁棒性继续前置：默认走“宽进严出”

本次 502 的根因已经通过架构调整规避：不再强依赖 `codex exec --output-schema`，而是走普通输出再做本地 JSON 提取。  
这条经验应该在 V3 固化为统一策略，而不是局部修补。

### 原则

- provider 输出尽量宽松接收
- 本地解析和 schema 校验尽量严格执行
- 错误诊断优先落在本地，不把 provider 的瞬时波动直接放大成整体失败

### V3 方向

- planner、handoff、summary 全部统一为“文本输出 + 本地抽取 + schema 校验”
- 对污染输出、前后缀废话、半截 JSON、重试恢复做统一抽象
- 结构化调用失败时，优先降级解析，不立刻走低质量 fallback

### 目标

- 上游 provider 抖动不再轻易把 plan / run 打回内置回退逻辑

---

## 七、summary 收敛升级：从“长摘要”到“决策面板”

V2 的 `summary` 已经能汇总很多信息，但对用户而言还不够像最终决策面板。

### V3 summary 至少要回答 7 个问题

1. 这次 run 是否完成
2. 哪些节点成功 / 失败
3. reviewer 是否放行
4. apply 是否发生、为什么发生或没发生
5. 哪些文件建议接收
6. 哪些文件必须人工复核
7. 哪些风险仍未关闭

### 建议输出结构

- `result_status`
- `review_gate`
- `apply_status`
- `accepted_files`
- `manual_review_files`
- `verified_capabilities`
- `blocked_verifications`
- `open_risks`
- `recommended_next_action`

### 目标

用户不需要再手动通读所有 handoff，直接看 summary 就能决定“收不收、收哪些、下一步干什么”。

---

## 八、`apply-mode none` 语义增强

本次 run 使用了 `--apply-mode none`，但它其实不应该只是“跳过应用”。  
在 V3 中，`none` 模式应该成为“**只做决策，不落地改动**”的完整工作流。

### V3 下的 `none` 模式应该输出

- 建议接收文件清单
- 建议拒绝文件清单
- 人工复核文件清单
- reviewer gate 结论
- 可复现验证摘要
- 下一步推荐命令

### 价值

这样 `none` 就不再只是“什么都不做”，而是“生成可执行审阅包”。

---

## 九、博客案例对应的 V3 策略模板

针对这次“简单 web 博客”案例，V3 应该默认收敛成下面这类策略：

### 推荐接收范围

- `Cargo.toml`
- `README.md`
- `src/blog/main.rs`
- `src/blog/content.rs`
- `src/blog/routes.rs`
- `src/blog/render.rs`
- `src/blog/assets/site.css`
- `content/blog/hello-codex-forge.md`

### 默认人工复核范围

- `src/main.rs`
- `src/cli.rs`
- `src/app.rs`
- `src/commander.rs`
- `src/orchestrator.rs`
- `tests/cli_integration.rs`
- 其他与博客目标无直接关系的 CLI 核心模块

### 推荐结论

- 博客 MVP 方案可放行
- 整包改动不应直接全收
- 应先做“交付瘦身”，再决定是否进入自动应用

## V3 实施阶段建议

## Phase 1：可信契约

目标：先让系统知道“什么叫越界”。

- 为 plan / execution graph 增加路径范围约束
- 引入 `ExecutionContract`
- 在 worker handoff 中补充契约对照字段

最小验证：

- 能在 session 工件中看见允许路径、禁止路径和范围偏差报告

## Phase 2：范围检测与部分接收

目标：让 apply 不再整包盲信。

- 引入 `ChangeTrustReport`
- 支持 apply 白名单和手动复核名单
- 在 `apply-mode none` 下输出建议接收清单

最小验证：

- 能在博客案例中自动识别核心文件与超范围文件

## Phase 3：环境感知验证

目标：让验证报告真正可信。

- 验证状态枚举化
- 自动识别环境阻塞并降级到等价验证
- summary 显式区分“失败”和“因环境未验证”

最小验证：

- 在沙箱端口受限环境下，报告中不再把 live smoke 缺失混成普通失败

## Phase 4：summary 决策面板

目标：让最终结论直接可执行。

- 重写 `summary` 收敛结构
- 输出建议接收 / 拒绝 / 人工复核清单
- 输出推荐下一步操作

最小验证：

- 用户只读 `summary.md` / `summary.json` 就能做接收决策

## Phase 5：worktree 与 provider 稳定性加固

目标：减少绕路与误报。

- 修复 reviewer / tester worktree 空壳问题
- 把本地 JSON 抽取策略推广到更多结构化环节
- 统一 provider 抖动、输出污染和重试恢复逻辑

最小验证：

- reviewer / tester 不再手动绕到 implementer 快照
- provider 短暂异常不再直接触发 plan fallback

---

## V3 验收标准

当以下条件成立时，可以认为 V3 达标：

- `plan` 与 `run` 的联动结果可解释、可追溯
- 系统能自动识别改动是否超出任务边界
- `apply-mode none` 能产出高质量审阅包，而不是简单跳过
- reviewer 放行不再等于整包默认可信
- 最终摘要能直接给出“建议接收 / 人工复核 / 剩余风险”
- 环境受限导致的未验证项会被单独标注
- worker worktree 的可用性问题基本消失
- provider 抖动不再轻易打穿结构化流程

---

## 一句话总结

V3 的核心不是“更多 agent”，而是把 codex-forge 从一个能调度多 agent 的 CLI，升级成一个**能对执行结果给出可信收敛判断的多 Agent 工程系统**。
